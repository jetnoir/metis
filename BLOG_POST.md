# How I Use Random Matrix Theory to Find Bugs in macOS Daemons

*17 April 2026 — Stuart Thomas*

---

When people ask how security researchers find vulnerabilities in closed-source software, they usually imagine one of two things: either a skilled human reading disassembly line by line, or a fuzzer throwing random inputs until something crashes. Both work. Both also scale badly. Manual review is expensive; unfocused fuzzing wastes compute on hardened code that almost certainly won't break.

This post describes a different approach — a six-stage automated pipeline that applies ideas from random matrix theory, statistical mechanics, and formal verification to triage macOS system daemons before a human reads a single line of assembly. The pipeline has produced multiple accepted Apple Security Bounty findings from a standing start, with no source code and no symbols. The full methodology is on [GitHub](https://github.com/).

---

## The problem

macOS ships with several hundred privileged daemons. Processes that run as root. Processes that speak the network. Processes that handle your files, manage security policy, and arbitrate between applications. Any one of them could contain a bug. But which one?

The honest answer is: you don't know. And there are too many to check all of them properly.

What you need is a triage mechanism — something that reads a compiled binary, no source code required, and tells you whether it's structurally normal or structurally suspicious. Not "this is definitely vulnerable," but "this one is unusual in a way that warrants closer attention."

That's what the toolchain does.

---

## The city analogy

Every program has a call graph: a map of which functions call which other functions. Some call graphs are clean — hierarchical, sparse, purpose-built. Others are tangled: one enormous function that everything passes through, recursive loops in unexpected places, structures that suggest code that grew organically over years without anyone stepping back to look at the whole.

Tangled structure correlates with bugs. Not because structural complexity causes bugs directly, but because code under pressure — code that accreted edge cases, was never properly refactored, handles protocol states nobody fully documented — tends to look tangled when you graph it. The clean code was probably written by someone who understood what they were building.

The first question the toolchain answers is: does this binary's call graph look like a normal program, or does it look like something unusual happened here?

---

## Random Matrix Theory: the mathematical core

The key tool is **Random Matrix Theory** (RMT). This is a branch of mathematics originally developed to study the energy levels of heavy atomic nuclei — physicists in the 1950s discovered that even without knowing the exact quantum mechanics, they could predict the *statistical distribution* of energy level spacings using random matrices. The same ideas turn out to apply to any complex system with many interacting parts.

For our purposes: given a call graph, we compute the **eigenvalue spectrum** of its adjacency matrix. The eigenvalues encode the structural properties of the graph — how information (and bugs) flow through the code.

The question is then: are these eigenvalues what you'd expect for a *random* graph of similar size and density? If yes, the binary is structurally normal. If no — if the eigenvalues deviate significantly from a random null distribution — something about this binary is unusual.

We compute three statistics from the spectrum:

- **Spectral radius** (ρ = max|λᵢ|): how strongly connected is the call graph?
- **Graph energy** (Σ|λᵢ|/N): total structural complexity per function
- **Eigenvalue entropy**: how evenly distributed is that complexity?

Each is z-scored against a null distribution generated by the **configuration model** (Bollobás 1980): random graphs with the same in/out degree sequence as the observed call graph. This is important — the naive choice of Marchenko-Pastur or Wigner semicircle law is wrong for call graphs, because they assume i.i.d. matrix entries, and real call graphs have power-law degree distributions. The configuration model controls for hub functions (the ones called by everything else) correctly.

A binary that deviates by more than 2 standard deviations from the null on any of the three metrics is flagged for deeper investigation.

This screen takes about 30 seconds per binary.

## What happens next

Flagged binaries go through a function-level triage. Here the tool uses **McCabe cyclomatic complexity** (M = E − N + 2, where E and N are control flow graph edges and nodes), **back-edge count** (a proxy for loop depth), and **eigenvector centrality** (which functions are hubs in the call graph).

Functions scoring high on a weighted combination of these metrics get promoted to manual or automated deep analysis.

The next stage matches the top-ranked functions against **vulnerability templates** — patterns that look like known bug classes at the call level:

```
recv/XPC/mach_msg receive  →  no bounds check  →  malloc with attacker-controlled size
```

A match doesn't confirm a bug. It flags a pattern worth examining.

Finally, for functions that survive all of this, symbolic execution via [angr](https://angr.io) attempts to find concrete inputs that reach sinks with attacker-controlled values. angr runs the binary with symbols — mathematical placeholders — instead of actual data, and asks its constraint solver whether there's a value that overflows a buffer, accesses freed memory, or hits a type confusion.

---

## The phase transition connection

The symbolic execution stage involves an idea from statistical physics that I find genuinely elegant. When angr explores a program, it accumulates **path constraints** — a growing set of logical conditions derived from the branches it resolves. These constraints are, from the solver's perspective, a SAT instance.

The 3-SAT problem has a **phase transition** at a critical clause-to-variable ratio α_c ≈ 4.267. Below that point, the problem is easy (almost certainly satisfiable). Above it, easy again (almost certainly unsatisfiable). Exactly at the transition: maximally hard — CDCL solvers take exponential time.

The **backbone fraction** — the proportion of variables that take the same value in every satisfying assignment — grows toward 1.0 as you approach the transition. It's a tractable proxy for solver hardness.

The C1 stage of the pipeline estimates backbone fraction for each execution path using assumption-based Z3 queries (not Survey Propagation, which the Tseitin CNF encoding destroys), and uses that estimate to prioritise which paths angr explores first. Low backbone fraction → probably easy → explore now. High backbone fraction → probably near the transition → defer.

Result: 60% reduction in states explored before finding a solution path, at 32ms per state probed.

## Two ideas that didn't survive

The pipeline originally had six components, C1 through C6. Two were killed during the design process — and I think documenting dead ends is as important as documenting what works.

**C4** was topological data analysis on control flow graphs: computing persistent homology of the basic block graph to capture its "shape." This is mathematically elegant. It is also operationally useless here, because loop structure and branching depth — exactly what TDA would detect — are already captured by back-edge count and cyclomatic complexity, computed in microseconds. TDA would take seconds per function via gudhi or ripser and add no discriminative signal.

**C5** was compressed sensing for fuzzing coverage bitmaps: projecting the 64K-entry AFL++ bitmap down to a small number of random linear measurements, then recovering the original. This requires the measurement matrix to satisfy the Restricted Isometry Property — which fails badly for coverage bitmaps, because most branches are taken the same way across most inputs. The sparsity structure is correlated, not random. The bitmap cannot be recovered from random projections.

The numbering was preserved rather than renumbered. The implemented pipeline jumps from C3 to C6. The ghost stages are a feature: anyone reading the codebase can see immediately that two ideas didn't survive contact with reality. A kill chain that killed C4 and C5.

## Findings

The pipeline has produced multiple accepted Apple Security Bounty findings. I'm not going to describe the specifics here — some are still under review, and Apple's responsible disclosure process deserves to run its course before public details appear.

What I can say: the RMT screen correctly identified the binaries worth looking at, and the function triage correctly identified the functions within those binaries. The findings were not ones I would have found with unfocused manual review of the same binaries in the same time.

## What it can't do

The honest version: it doesn't always work.

angr hits memory limits on large binaries (roughly 3.3 MB Mach-O limit on current hardware). Compiler optimisation — specifically, constant-folding arithmetic into load immediate offsets — hides patterns that the VEX IR scanner expects. Some bugs are architectural: no static analysis finds a missing entitlement check without understanding the XPC protocol semantics. arm64e pointer authentication (PAC) blocks function pointer hijack even when an overflow primitive exists, forcing a pivot to state corruption instead.

The pipeline is a triage tool. It narrows 300 functions down to 3. What you do with those 3 is still a human problem.
